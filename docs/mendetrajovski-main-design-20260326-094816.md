# Design: ContextGit — Developer-Led Growth Strategy

Generated by /office-hours on 2026-03-26
Branch: main
Repo: MendeTr/contextgit
Status: APPROVED
Mode: Startup

## Problem Statement

AI coding agents (Claude Code, Cursor, Windsurf) suffer from session amnesia — they lose
context between sessions, can't resume long tasks, and conflict with each other when run
in parallel. The current workaround is CLAUDE.md-style manual notes: slow, stale by end
of session, and completely broken for multi-agent workflows.

ContextGit solves this by making the commit the unit of memory — git-native context that
persists across sessions, agents, and branches.

## Demand Evidence

- **Primary evidence:** The founder is the most frustrated user — uses ContextGit in
  ContextGit's own development (the CLAUDE.md of this repo is a working proof of concept).
  All three amnesia variants hit constantly: lost work between sessions, inability to
  coordinate agents, and progress loss on long multi-week projects.
- **Secondary evidence:** The problem is actively discussed in developer communities online.
  Not named individuals yet — no one has directly asked for this product. That's the gap.

## Status Quo

Developers managing AI agents use CLAUDE.md-style manual notes files. Cost:
1. Agents act on stale context, making changes that conflict with what's already built
2. Files rot immediately — accurate at session start, wrong by session end
3. Completely falls apart with multiple agents — no shared state, guaranteed conflicts

The incumbent is duct tape. Everyone is maintaining their own version of CLAUDE.md.

## Target User & Narrowest Wedge

**Target:** Solo developer building seriously with AI agents — someone spending 4+ hours/day
with Claude Code or Cursor on real projects. Not vibe coding a weekend toy. Building
something they intend to ship.

**Their failure:** Losing progress on long projects. Context resets kill momentum.
Multi-week builds become impossible when each session restarts from zero.

**Narrowest wedge:** The MCP server that makes `project_memory_load` and
`project_memory_save` available in Claude Code with zero configuration. One `mcp add`
command. The developer doesn't need to understand ContextGit's internals — they just
need agents that remember.

## Constraints

- Pre-product: founder is the only real user
- No paying customers, no user interviews beyond self
- Distribution: MCP is the right entry, but CLI + API are also needed for standalone use
- Must validate demand beyond personal use before building more infrastructure

## Premises

1. **Solo developer first, enterprise long-term.** The beachhead is individual developers
   who feel the pain directly. Enterprise is the long-term market. ✓ Agreed.

2. **Git-native is the wedge, not the moat.** Mem0/Zep cannot replicate a commit-as-memory
   architecture without rebuilding from scratch — that's a strong wedge. But the long-term
   moat is still TBD (network effects? data? workflow lock-in?). ✓ Agreed with nuance.

3. **MCP is the right entry point, but needs standalone CLI + API too.** MCP distribution
   reaches the most motivated users with zero friction. But the product must work without
   a specific MCP host. ✓ Agreed.

## Approaches Considered

### Approach A: Developer-Led Growth (CHOSEN)
- Summary: Polish the MCP install flow, simplify onboarding, ship a compelling README.
  Grow through GitHub stars and word-of-mouth. Hit 50 real installs before building
  anything else. Measure demand signals, not features.
- Effort: S (2-3 weeks)
- Risk: Low
- Pros: Tests real demand fast; no new architecture; compounding word-of-mouth
- Cons: Solo devs may not pay; slow to reach enterprise; moat is thin until usage data exists
- Reuses: MCP server + CLI already built

### Approach B: Multi-Agent Coordination Platform
- Summary: Build real-time agent coordination — shared context, conflict detection, task
  routing. The live claims + Supabase work points here.
- Effort: L (1-2 months)
- Risk: Med (market timing, enterprise sales cycle)
- Reuses: Supabase claims + LocalStore already built

### Approach C: Managed Cloud + Registry
- Summary: contextgit.io — hosted memory, public context templates, clear monetization.
- Effort: XL (3-6 months)
- Risk: High (platform plays need critical mass)

## Recommended Approach

**Approach A: Developer-Led Growth.**

The product is functional. The founder uses it constantly. The gap is not features —
it's that nobody else knows it exists. The right move is distribution, not architecture.

Specific execution:
1. Write a README that explains the amnesia problem in one paragraph and shows the
   `mcp add` command in the second paragraph. No jargon.
2. Post in 3-5 developer communities where agent amnesia is discussed (Claude Discord,
   Cursor forums, r/ClaudeAI, dev.to). Format: one paragraph on the amnesia problem, one
   paragraph on the tool, link to repo. No marketing language.
3. Track: GitHub stars, `mcp add` attempts, return users (sessions > 1).
4. Talk to the first 10 people who install it. Not a survey — a 15-minute call.
5. Only after 50 real installs, revisit the moat question.

## Open Questions

- What IS the long-term moat if git-native is just a wedge? Network effects among teams?
  A data layer that learns from usage patterns? A protocol that other agents adopt?
- Is the solo developer willing to pay? What's the pricing model?
- Does ContextGit need to work without Claude Code specifically — i.e., is it Cursor/
  Windsurf compatible today?

## Success Criteria

- 50 real installs — proxy: npm weekly download floor + GitHub issue self-reports
  (no telemetry; acknowledge this is imprecise)
- 5 developers who confirm continued use via DM or Discord across 2+ calendar days
  (honor-system; no telemetry to verify `project_memory_save` call counts)
- 1 developer who would be "very disappointed" if it disappeared (Ellis's PMF survey)
- The first time someone finds a bug and files an issue (they care enough to report it)

Return-user tracking: no telemetry in the product. Measure via GitHub issue self-reports,
Discord check-ins, or a simple "are you still using it?" follow-up DM to the first 10 installers.
Explicit opt-in only — no silent instrumentation.

After 50 installs: revisit pricing model and moat question. After 100 installs: consider
whether multi-agent coordination (Approach B) is worth starting.

## Distribution Plan

**Current:** MCP server (npm package), CLI (npm package), GitHub repo.
**Gap:** The install experience is not validated. No one other than the founder has
attempted onboarding from a cold start.

**Install success definition:** A cold-start install succeeds when a developer who has
never seen the project can run the install command, get a working MCP tool in Claude Code,
and call `project_memory_save` — without asking for help. Target: macOS and Linux,
Node 20+. Windows is out of scope for the initial wedge; revisit if demand appears.

**Next:**
- Test the README cold: give it to someone who hasn't seen the project and watch them try
  to install it without help. (Find them via Claude Discord or a developer friend.)
- Test MCP server against Cursor in week 1. If compatible: include in distribution posts.
  If incompatible: post anyway with explicit "Claude Code only" scope; revisit Cursor later.

**Later (after 50 installs):**
- GitHub Actions: automated npm publish on version bump (currently manual).

## Dependencies

- None blocking. All infrastructure is already built.
- The Supabase remote sync is optional — local SQLite is sufficient for the solo dev wedge.

## The Assignment

**This week:** Share ContextGit in one community where you've personally seen the agent
amnesia problem discussed. Not a pitch — a post that describes the problem and links the
repo. Count how many people click through, star it, or ask questions. That number is
your first real demand signal beyond yourself.

Don't build anything new until you have that number.
