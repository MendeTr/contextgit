# ContextGit — Delta Spec: Connectivity & Save-Rhythm Fixes

**Date:** 2026-05-21
**Status:** Ready for implementation
**Target version:** 0.1.x (next patch release)
**Scope:** `packages/core` (claude-md-generator), `packages/cli` (init-helpers)

---

## Why this spec exists

A Claude Code usage audit of a real session found five problems. Three of them are
bugs — fixable now, small, no design work needed. This spec covers those three. The
other two (saves duplicate git log; threads accumulate without decay) are feature
scope and live in `02_ContextGit_DELTA_granularity.md`.

**This patch must ship before the granularity spec.** Until Bug 1 is fixed, the save
half of ContextGit is physically disconnected on every project that had a `CLAUDE.md`
before installing ContextGit — which is most real projects. No measurement of any
later work is meaningful until this ships and a fresh audit runs against a connected
install.

---

## Bug 1 — `@CLAUDE.contextgit.md` include is never wired

### Symptom

On a project with a pre-existing `CLAUDE.md`, every `project_memory_save` prints:

> "CLAUDE.md already exists and was not generated by ContextGit. Context written to
> CLAUDE.contextgit.md instead. Add '@CLAUDE.contextgit.md' as the first line of your
> CLAUDE.md for auto-synced project context."

The sidecar `CLAUDE.contextgit.md` is written, but `CLAUDE.md` never includes it, so
no future session loads it. Saves are write-only. The audit called this "write-only
theater" — correctly.

### Root cause

`packages/core/src/claude-md-generator.ts`, method `write()`. Three branches: no
CLAUDE.md → create it; CLAUDE.md is ContextGit-owned → overwrite it; CLAUDE.md is
manually maintained → write the sidecar and **return a warning string telling the
human to wire the include by hand.** No branch ever edits `CLAUDE.md`. The auto-wire
behavior was specced (it is in the project's design notes) but never implemented.

Compounding cause across packages: `init-helpers.ts` `writeClaude()` *appends* the
ContextGit fragment to an existing `CLAUDE.md`. So after `init`, `CLAUDE.md` is
"manually-maintained + our fragment" and does not start with the auto-gen marker —
which guarantees `write()` falls into branch 3 forever. The two files disagree on
what `CLAUDE.md` is: `init-helpers` treats it as hand-maintained-and-appended;
`claude-md-generator` treats it as a build artifact it owns. The seam between those
two models is exactly where the loop drops.

### Fix

In `claude-md-generator.ts` `write()`, branch 3 (manually-maintained CLAUDE.md):
after writing `CLAUDE.contextgit.md`, also edit `CLAUDE.md` to wire the include.

1. Read `CLAUDE.md`.
2. If `@CLAUDE.contextgit.md` already appears as a line, do nothing further.
3. Otherwise prepend `@CLAUDE.contextgit.md` as the **first line**, followed by a
   blank line, preserving all existing content below it.
4. Write `CLAUDE.md` back.

Idempotent — running twice does not duplicate the line. ~6 lines of code.

The returned `WriteResult.warning` changes from an instruction-to-the-human into a
success note: "Context written to CLAUDE.contextgit.md and wired into CLAUDE.md via
@-include."

### Do NOT

Do not make the generator take full ownership of a hand-written `CLAUDE.md`. People
legitimately want a hand-maintained `CLAUDE.md` plus a synced sidecar — that is the
entire reason the sidecar exists. Wire the include; do not overwrite the file.

### Tests

In `claude-md-generator.test.ts`:

1. Manually-maintained `CLAUDE.md` → after `write()`, first line is
   `@CLAUDE.contextgit.md` and original content is preserved below.
2. `write()` called twice → include line appears exactly once.
3. `CLAUDE.md` that already has the include line → content unchanged, no duplicate.
4. No `CLAUDE.md` → branch 1 unchanged (creates ContextGit-owned file, no include).
5. ContextGit-owned `CLAUDE.md` → branch 2 unchanged (overwritten, no include).

## Bug 2 — saves are never re-read (consequence of Bug 1)

### Symptom

The audit: 5 detailed saves written in one session, none ever consulted.
"Information went out, nothing came back."

### Root cause

Direct consequence of Bug 1. The next session reads `CLAUDE.md`; the saves are in
`CLAUDE.contextgit.md`; the two were never linked.

### Fix

No separate work. Fixing Bug 1 resolves this. Listed explicitly so the audit's
five-point list maps cleanly and nothing is assumed handled silently.

### Verification

Part of Bug 1's validation gate (see Validation Gate below) — a second session must
demonstrably read what the first session saved.

## Bug 3 — the save rhythm manufactures noise

### Symptom

The audit: the "every git commit = context commit" rule produced ~30 saves' worth of
pressure across Plan 1A; the agent overruled it down to 5. Most of what the rule
asks for paraphrases the git commit message — noise. The agent ignored a MANDATORY
instruction *because the instruction was wrong*.

### Root cause

The rule is asserted in **three** places in `packages/cli/src/lib/init-helpers.ts`,
all regenerated on every fresh `init`:

- `CLAUDE_MD_FRAGMENT` — "Every git commit = immediate context commit ... Do not
  proceed to the next task until both are done."
- `CONTEXT_COMMIT_SKILL` (the skill description) — "MANDATORY ... after EVERY
  completed task and EVERY git commit ... Do not batch."
- `CONTEXTGIT_HOOKS.PostToolUse` — matcher `Bash`, condition `Bash(git commit*)`,
  injects "MANDATORY: Call project_memory_save NOW."

This contradicts the project's own prior fix (context_commit was already identified
as being called too frequently). That fix changed *behavior in one session* but left
all three *enforcement surfaces* intact, so the bug regenerates on every install.

### Fix

Replace the rule in all three surfaces with the wording below. This exact text is
shared with `02_ContextGit_DELTA_granularity.md` — keep them identical.

**Remove:** "Every git commit = immediate context commit. Do not batch. Do not
proceed until both are done."

**Replace with:**

- Call `project_memory_save` at **session end**, always — a focused summary plus the
  3–5 genuinely open threads for the next session.
- Call `project_memory_save` mid-session **only when project state changes in a way
  git does not capture**: a decision made, an approach abandoned, a thread opened or
  closed, scope changed (`replan:` prefix), an architectural choice.
- Do **not** save merely because a git commit happened. Git is the record of what
  changed. A context save that only paraphrases a commit message is noise.
- `project_memory_branch` before risky exploration — unchanged, still encouraged.

**`CONTEXTGIT_HOOKS` — remove the `PostToolUse` block entirely.** It is the mechanism
that fires on every `git commit`; a gentler message on the same trigger still fires
every commit. The `SessionStart` hook (mandatory `project_memory_load`) **stays** —
the audit confirmed the load is the highest-value call and should remain enforced.

### Tests

In `init-helpers.test.ts` (or wherever `patchClaudeSettings` is covered):

1. `patchClaudeSettings` output contains a `SessionStart` hook.
2. `patchClaudeSettings` output contains **no** `PostToolUse` hook.
3. `CLAUDE_MD_FRAGMENT` does not contain the string "Every git commit".
4. `CONTEXT_COMMIT_SKILL` description does not contain "Do not batch".

## Validation gate

This patch is complete when:

1. On a project with a pre-existing `CLAUDE.md`, `project_memory_save` wires
   `@CLAUDE.contextgit.md` into `CLAUDE.md` automatically — verified by inspecting
   the file.
2. A second Claude Code session in that project demonstrably loads what the first
   session saved (open it, confirm the agent has prior-session context without being
   told).
3. A fresh `init` produces a `.claude/settings.json` with a `SessionStart` hook and
   no `PostToolUse` hook.
4. A re-run of the Claude Code audit ("is ContextGit helpful, honest answer") against
   this connected install no longer contains "write-only theater."

Result 4 is the baseline against which `02_ContextGit_DELTA_granularity.md` is
measured. Do not begin the granularity spec until this patch is published and audited.

---

## What does NOT change

- The MCP server surface (`packages/mcp`) — unchanged.
- The project DB, config, git-hooks install logic — unchanged.
- `project_memory_save` / `_load` / `_branch` tool signatures — unchanged. Only the
  *guidance text* on when to call `save` changes.
- Thread storage and the open-thread accumulation problem — out of scope here; that
  is `02_ContextGit_DELTA_granularity.md`.
